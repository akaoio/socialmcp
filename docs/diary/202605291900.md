# 2026-05-29 — User namespace + signed writes (ZEN security)

## Vấn đề

Session trước dùng public namespace `socialmcp/cmd/<id>` và `socialmcp/res/<id>`. Đây là **public graph** — bất kỳ peer nào kết nối vào relay đều có thể:
- Inject lệnh giả: `zen.get('socialmcp').get('cmd').get('999').put('{"platform":"facebook",...}')`
- Giả mạo response: `zen.get('socialmcp').get('res').get('1').put('{"ok":true}')`

ZEN không có cơ chế nào ngăn điều này với public namespace vì `check.any` chỉ block nếu `at.opt.secure` được bật — mặc định là open.

## Giải pháp: User namespace `~<pub>/...`

ZEN có security middleware cho **user-signed namespace**. Khi soul bắt đầu bằng `~<pub>/`, ZEN chạy `check.pub`:

```js
// zen.js:3438
} else if ((tmp = settings.pub(soul))) {
  ctx.pub = tmp;
  pipeline.push(check.pipe.pub);
}
```

`settings.pub(soul)` extract `pub` từ soul dạng `~<pub>/path`. `check.pub` sau đó:

```js
if (
  ((user && user.is) || authenticator) &&
  upub &&
  !raw["~"] &&
  pub === upub   // soul's pub must match authenticator's pub
) {
  check.auth(msg, no, authenticator, function (data) { ... });
}
```

`check.auth` ký data bằng `authenticator.priv`, nhúng signature vào message. Nếu không có authenticator khớp với pub trong soul → `no("Unverified data.")`.

## Cách put đúng với authenticator

Trace đầy đủ từ API đến security middleware:

1. `zen.get('~'+pair.pub).get('cmd').get(id).put(value, null, { authenticator: pair })`
2. `chain.put(data, cb, opt)` → `options(opt)` → `{ authenticator: pair }` (shallow copy)
3. `as.ran = function(a) { ran(a, opt); }` — opt captured in closure
4. `ran(as, opt)` → outgoing message: `{ put: graph, opt: opt, ... }`
5. Trong `put()` handler (line 824): `ctx.msg = msg`
6. `check.$zen(msg, user, pub)`:
   - `ctx = scope.opt || (scope.msg || {}).opt || {}` → `msg.opt = { authenticator: pair }`
   - `initZenOpt(msg, ctx)` → `o = { authenticator: pair }`
   - `authenticator = o.authenticator = pair`
7. `check.auth(msg, no, pair, done)` → ký bằng `pair.priv`, verify `pair.pub === soul.pub` ✓

Chữ ký SAI sẽ bị relay reject trước khi broadcast đến peer khác.

## Keypair derivation (cả hai đầu)

Cả server lẫn extension derive cùng keypair từ cùng secret:

```js
const seed = await ZEN.hash(secret, null, null, { name: 'SHA-256', encode: 'base62' });
const pair = await ZEN.pair(null, { seed });
// pair = { pub: '...45 chars...', priv: '...', curve: 'secp256k1' }
```

- Server: secret từ `SOCIALMCP_SECRET` env var
- Extension: secret từ `chrome.storage.local.get(['secret'])`
- Cả hai fallback về `'socialmcp-local-default'` nếu chưa set

## Thay đổi code

### bridge.js
- Không còn `const NS = 'socialmcp'`
- Thêm `this.pair`, `this.ready`, `_initpair()` async method
- `start()` gọi `_initpair()`, log thêm `(pub: <pubkey>)`
- `send()` trở thành `async`: `await this.ready` trước khi dùng pair
- Namespace: `const ns = '~' + pair.pub`
- Mọi `.put()`: thêm `null, auth` → `put(value, null, { authenticator: pair })`
- Cleanup null writes cũng cần auth: `cmd.put(null, null, auth)`

### background.js
- Không còn `const NS = 'socialmcp'`
- Thêm `getsecret()` — đọc `chrome.storage.local`
- `connect()` trở thành `async` — await `getsecret()`, `ZEN.hash()`, `ZEN.pair()`
- Namespace: `const ns = '~' + pair.pub`
- Mọi `.put()` trong listener: thêm `null, auth`
- `keepalive` alarm: `.catch(() => { zen = null; })` để reconnect nếu có lỗi

## Tradeoffs

- Server `send()` bây giờ là `async` vì phải `await this.ready`. `index.js` không cần thay đổi vì nó đã `await bridge.send(...)`.
- `connect()` trong extension là `async` nhưng alarm handler không cần `await` — fire and forget, nếu fail thì `zen = null` để lần sau retry.
- Nếu hai phía dùng secret khác nhau → pub khác nhau → extension lắng nghe sai namespace → timeout. Debug bằng cách so sánh pub được log ra ở server.
