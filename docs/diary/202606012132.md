# 2026-06-01 — Tests rewritten to use full MCP JSON-RPC pipeline

## What changed

All integration tests now exercise the exact same path an AI agent uses:

```
test stdin (JSON-RPC tools/call)
  → MCP server (src/server/index.js)
  → bridge.send() → HTTP relay localhost:8765
  → peer.js GET /job → dispatch → DOM
  → POST /result/:id → stdout
```

Previously tests bypassed MCP entirely — they called `window.dispatch()` directly
from a relay page context, skipping the server, bridge, and peer layers entirely.

## Files changed

- **`src/server/bridge/bridge.js`**:
  - Added `SOCIALMCP_NO_AUTOLAUNCH=1` env var to disable auto-launch during tests
  - Added `GET /ready` endpoint: returns `{ connected: true/false }` for readiness polling
  - Changed port from `8420` → `8765` (8420 was conflicting with another local service)

- **`src/browser/background/peer.js`**: updated base URL to `:8765`

- **`src/server/schema.js`**: added `boolean()` type

- **`src/server/index.js`**: added `dryrun` parameter to `post` tool schema (was supported by extension but undocumented in MCP)

- **`tests/mcpclient.js`** (new): MCP stdio client helper
  - `startmcp()` — spawns `node src/server/index.js`, sends `initialize` handshake
  - `call(name, args)` — sends `tools/call` via stdin, reads from stdout, parses result
  - `waitforpeer(timeout)` — polls `/ready` until extension peer has connected
  - Fail-fast: rejects all pending on server exit

- **`tests/mcp.spec.js`** (new): protocol-layer tests (no extension needed)
  - `initialize` returns correct protocol version + server info
  - `tools/list` returns all 6 tools
  - `post` tool advertises `dryrun` param
  - All tools have `platform` enum with correct values

- **`tests/extension.spec.js`**: rewritten as extension smoke + bridge connectivity check
  - Still checks service worker URL
  - Now also starts MCP server and verifies peer connects

- **`tests/debug.spec.js`**, **`facebook.spec.js`**, **`post.spec.js`**: all rewritten to use `mcp.call()`

## Tradeoffs

- Each test file spawns its own MCP server subprocess (port 8765). Since tests run serially, this is safe — no port conflicts.
- `waitforpeer()` polls `/ready` instead of a fixed sleep — reliable readiness detection.
- Killing the test server (SIGTERM) between files means the port is always free for the next file.
