# 2026-05-30 22:30 — Drift audit + stabilization

## Findings

Deep audit of the codebase produced one critical and several medium drifts.

### CRITICAL
- **ZEN peer not implemented in `background/`**. `src/server/bridge.js` writes commands to `~<pub>/cmd/<id>` and awaits responses on `~<pub>/res/<id>`, but `src/browser/background/index.js` only listens to `chrome.runtime.onMessage` (dashboard → background). The MCP-server-to-extension path is non-functional; only the dashboard can drive actions today. Documented as a known gap in `docs/plugin-dev-guide.md` and `.github/copilot-instructions.md` rather than silently re-asserting the aspirational architecture.

### MEDIUM (fixed in this session)
1. **Platform list duplication.** `build.js` hard-coded `PLATFORMS = ['facebook']`. Replaced with `discoverplatforms()` that scans `src/browser/platform/*/plugin.js`.
2. **Manifest duplication.** `manifest.json` `content_scripts` + `host_permissions` hard-coded the FB host. `build.js` now regenerates the manifest from each plugin's `hosts.js` (regex-parsed). Source manifest still carries dev-mode entries.
3. **FB-specific magic in generic dispatch.** `background/dispatch.js` had a `params.page_url ?? params._url` heuristic that auto-navigated. Removed — generic dispatch is now strictly `findtab → sendmessage`. Platform-specific multi-step flows must live in the plugin's own `background/<action>.js`.
4. **Public action / MCP tool name mismatch.** FB plugin exposed `background.postpage` while the MCP `post` tool would call action `'post'`. Renamed: plugin now declares `background.post` and `background.scan`. Internal content-script action names (`postpage`, `switchpage`, `getpages`) stay private. Dashboard updated to call public action names.
5. **`platform/facebook/background/dispatch.js` rename.** Replaced with one-function-per-file: `post.js` and `scan.js`. Plugin manifest imports both.
6. **`chrome.storage.local` key collision risk.** `fb_pages` had no plugin scope. Migrated to `'facebook:pages'`. Convention codified in dev guide.
7. **Manifest host wildcard.** `https://www.facebook.com/*` → `https://*.facebook.com/*` (covers `m.`, `business.`).

### MINOR (documented as conventions, not auto-fixed)
- DOM ids in plugin panels must be `<plugin-id>-...` prefixed. Currently followed by FB (`fb-...`). Locked in as a hard rule in the dev guide.
- `chrome.storage.local` keys must use the `<plugin-id>:<key>` format.
- Each feature folder owns its own `selectors.js`; no platform-root shared selectors.

## New artifact

`docs/plugin-dev-guide.md` — single source of truth for plugin/feature dev. Contains:
- mental model + folder layout
- 7 hard rules
- step-by-step "add a platform" and "add a feature"
- ZEN gap documented explicitly
- drift checklist for PRs

`.github/copilot-instructions.md` and `README.md` updated to reflect the new conventions and link to the guide.

## Not done (intentionally)

- ZEN peer implementation in `background/`. Scope creep — separate feature work. Surfaced loudly so it cannot be missed when next picked up.
- Source manifest auto-regeneration for dev mode. Current workflow: edit source manifest by hand for dev, build script regenerates for prod. Acceptable until a `npm run dev` watcher is introduced.
